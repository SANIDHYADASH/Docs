The Problem

⏱️

Manual Trace Analysis

Glowroot produces rich trace data, but reviewing exports manually is slow and error-prone — analysts must sift through nested call trees, SQL queries, and timer breakdowns.

🔍

Hidden Anomalies

Critical patterns like N+1 queries, connection pool exhaustion, and subtle CPU bottlenecks are easy to miss without systematic, automated checks.

📊

No Risk Quantification

Raw traces don't provide a single risk score or prioritised finding list — teams lack a clear "how bad is it?" answer for go/no-go decisions.

🔄

Slow Feedback Loop

Without CI/CD integration, performance regressions are caught late — often only after production incidents, not at the pull-request or staging phase.

Two-Layer Analysis Engine

⚙️

Layer 1 — Rule Engine

Always On Deterministic
  • 20+ detection patterns
  • No external dependencies
  • Response time, SQL, error, JVM, concurrency checks
  • Pattern matching on N+1 queries, SELECT *, leading wildcards
  • Timer & entry bottleneck analysis
  • Configurable thresholds via config.yaml
🧠

Layer 2 — LLM Analyzer

Optional AI-Powered
  • Root-cause analysis & scaling predictions
  • Architecture concerns & concurrency issues
  • Java / Spring Boot code fix suggestions
  • Structured prompt with trace summary + Layer 1 findings
  • Providers: GitHub Copilot, OpenAI, Azure OpenAI
  • Graceful fallback: provider: "none"

When no LLM is configured the pipeline continues with rule-based findings only — zero degradation in core detection.

Detection Patterns (Layer 1)

# Pattern Severity Trigger
1 Critical response time CRITICAL Duration > 10 s
2 Slow response time HIGH Duration > 3 s
3 Transaction errors CRITICAL Timeout, OOM, deadlock, NPE, SQL, pool exhaustion
4 Slow SQL query HIGH Query > 1 s
5 Very slow SQL query CRITICAL Query > 5 s
6 Excessive data fetch HIGH Rows > 10,000
7 N+1 query problem CRITICAL Same query ≥ 10 times
8 Deep call stack MEDIUM Depth > 50
9 Excessive trace entries HIGH Entries > 1,000
10 Timer bottleneck HIGH Single timer > 80% of duration
11 Single entry bottleneck HIGH One entry > 80% of duration
12 Connection pool pressure HIGH Multiple short waits on pool
13 I/O-bound transaction MEDIUM CPU < 10% of wall time
14 CPU-intensive transaction HIGH CPU > 90% of wall time
15 SELECT * anti-pattern MEDIUM Detected in SQL text
16 Leading wildcard LIKE MEDIUM LIKE '%...' in SQL
17 Missing index inference MEDIUM Full table scan indicators

09 / 12

CI/CD Pipeline Integration

Block deployments automatically when risk score exceeds threshold.

GitHub Actions Example

# .github/workflows/perf-gate.yml - name: Analyze Glowroot Traces run: | RESULT=$(curl -s -X POST \ http://predictor:5050/api/analyze/json \ -F "file=@glowroot-trace.html") RISK=$(echo $RESULT | jq '.risk_score') if [ $(echo "$RISK > 70" | bc) -eq 1 ]; then echo "❌ CRITICAL risk: $RISK" exit 1 fi

Gate Thresholds

Score 0–24 — Deploy freely
Score 25–44 — Deploy with warnings logged
Score 45–69 — Require manual approval
Score 70–100 — Block deployment automatically

Getting Started

Prerequisites

  • Python 3.10+
  • Gemini / OpenAI / Azure OpenAI API key

Install & Run

# Install dependencies cd ai-predictor pip install -r requirements.txt # Analyze a trace (HTML report) python main.py analyze trace.html # Analyze with JSON output python main.py analyze trace.html --format json # Batch analyze a directory python main.py analyze ./exports/ # Start the web dashboard python main.py dashboard # Live monitoring python main.py monitor

Configuration Highlights

# config.yaml thresholds: response_time: warning_ms: 3000 critical_ms: 10000 queries: slow_query_ms: 1000 n_plus_one_threshold: 10 model: provider: "github_copilot" github_model: "gpt-4o" dashboard: host: "0.0.0.0" port: 5050 logging: level: "INFO" file: "./logs/predictor.log"
